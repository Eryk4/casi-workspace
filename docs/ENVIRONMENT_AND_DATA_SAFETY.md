# Environment And Data Safety

Last checked: 2026-07-03

CASI Workspace is a real business application. Local seeded data is only a safe working sandbox used for development, verification, and pre-production review. The sandbox is not the product goal and must not be confused with a production or customer data environment.

## Environment Types

Recommended environment model:

| Environment | Purpose | Data | Reset allowed |
| --- | --- | --- | --- |
| Local sandbox | Local development and verification | Fictional working data generated by seed scripts | Yes, after confirming the database is local and backed up |
| Dev / test | Shared engineering checks | Synthetic or anonymized data | Only through explicit test procedures |
| Staging | Release validation before production | Production-like but controlled data | No destructive reset without written approval and backup |
| Production | Real customer usage | Real business and personal data | Never through local reset/reseed scripts |

## Current Local Database

The current local default uses SQLite under the repository `data/` directory. The expected local database path is:

```text
data/invoice_ops.sqlite3
```

Local configuration can change this through environment files such as:

```text
.env.local
data/local.env
```

Before any destructive operation, inspect the effective configuration and confirm that the app is not using:

- `DATABASE_URL` pointing to a remote database,
- `INVOICE_DATABASE_URL` pointing to a remote database,
- PostgreSQL,
- Railway,
- DigitalOcean,
- any shared or customer-owned database.

If there is uncertainty, stop and do not reset.

## Reset And Reseed Rules

Reset/reseed is allowed only when all of these are true:

- the database is confirmed to be local SQLite,
- neither `DATABASE_URL` nor `INVOICE_DATABASE_URL` is active,
- the SQLite path is under the repository `data/` directory,
- the environment explicitly allows local sandbox reset through `CASI_ALLOW_LOCAL_SANDBOX_RESET=1`,
- the data is fictional sandbox data,
- a backup was created immediately before the operation,
- the operation is needed for development or verification,
- the user explicitly asked for it or approved it.

Reset/reseed is forbidden when:

- the database contains real business data,
- the app uses a remote database,
- the environment is staging or production,
- the database owner is unclear,
- the operation would affect customer documents, invoices, CRM data, payments, or audit history.

## Backup Before Destructive Work

Before a local reset:

1. Identify the active database path.
2. Create a timestamped backup of the SQLite file.
3. Do not delete the `data/` directory unless the task explicitly requires it.
4. If file storage is relevant, back it up or leave it untouched.
5. Record the reset and seed commands in the task summary.

Recommended local backup naming:

```text
data/backup/reset/invoice_ops_YYYYMMDD_HHMMSS.sqlite3
```

The current `python run.py --reset` guardrail creates this backup automatically when the local SQLite file already exists. If backup creation fails, reset is blocked.

## Safe Local Sandbox Reset Command

Use this only after confirming that the active database is the local sandbox SQLite under `data/`.

PowerShell:

```powershell
$env:CASI_ALLOW_LOCAL_SANDBOX_RESET = "1"
python run.py --reset
Remove-Item Env:\CASI_ALLOW_LOCAL_SANDBOX_RESET
```

Command Prompt:

```cmd
set CASI_ALLOW_LOCAL_SANDBOX_RESET=1
python run.py --reset
set CASI_ALLOW_LOCAL_SANDBOX_RESET=
```

The reset is blocked when:

- `CASI_ALLOW_LOCAL_SANDBOX_RESET=1` is missing,
- `DATABASE_URL` is set,
- `INVOICE_DATABASE_URL` is set,
- the configured engine is not SQLite,
- the SQLite path is outside `data/`,
- the path looks like a non-local or production path.

## Seed Scripts

The technical file `app/demo_seed.py` currently seeds the local working sandbox. Its historical name is technical debt and does not mean CASI is a demo product.

Interpretation:

- CASI application = real operational business system.
- Seed data = fictional local sandbox data.
- Seed script = a technical mechanism for preparing safe data for development and verification.

Do not rename seed files casually. Renaming can affect tests, scripts, and operational shortcuts. If renaming is desired later, do it as a separate migration-style cleanup with tests.

## What Codex Must Not Do On Real Data

Codex must not run reset/reseed, destructive scripts, bulk data edits, storage cleanup, or migration reset commands against data that may be real.

Codex must also avoid:

- deleting customer files,
- clearing invoice or document history,
- changing production configuration,
- replacing production secrets,
- running local sandbox scripts against staging or production,
- logging secrets, tokens, local paths, or raw sensitive payloads.

## Git And Local Data

Local sandbox data, real customer data, local backups, generated storage files, logs, and private environment files must stay outside Git history.

Do not commit:

- `.env`, `.env.local`, or any private `.env.*` file,
- `data/*.sqlite3` or any nested SQLite database,
- `data/backup/` or `data/backups/`,
- `data/magazyn/`,
- generated document/OCR text under local `data/`,
- local server logs,
- `.next/`, `node_modules/`, Python virtual environments, or cache directories.

Allowed env templates:

- `.env.example`
- `.env.local.example`

These templates must contain placeholders only. They must not contain real IMAP passwords, OAuth secrets, API keys, database URLs, S3 credentials, tokens, or customer data.

When moving the project to another computer:

1. Move source code through Git.
2. Copy private env files manually through a secure channel, not through commits.
3. Copy local sandbox databases or storage only if explicitly needed, and keep them outside Git.
4. For real data, use a proper backup/restore process outside repository history.

If sensitive data was already added to Git tracking, do not delete the local data. Remove it from the Git index with a deliberate command such as:

```bash
git rm --cached <path>
```

If the sensitive data was committed, repository history cleanup may be required before sharing the repository.

## Known Risk

The current local reset flow is powerful and convenient for sandbox work. The risk is accidental use against a non-sandbox database if environment variables point to a shared or remote system.

Recommended next guardrail:

- add a startup/status surface that clearly shows whether the current process is using local sandbox data, staging data, or production data.
